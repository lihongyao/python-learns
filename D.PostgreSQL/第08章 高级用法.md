# 连接 `JOIN`

@See https://www.runoob.com/postgresql/postgresql-join.html

PostgreSQL JOIN 子句用于把来自两个或多个表的行结合起来，基于这些表之间的共同字段。

在 PostgreSQL 中，JOIN 有五种连接类型：

- CROSS JOIN ：交叉连接
- INNER JOIN：内连接
- LEFT OUTER JOIN：左外连接
- RIGHT OUTER JOIN：右外连接
- FULL OUTER JOIN：全外连接

接下来让我们创建两张表 **COMPANY** 和 **DEPARTMENT**。

**实例**

创建 COMPANY 表，sql 指令如下：

```postgresql
DROP TABLE IF EXISTS company;

CREATE TABLE company(
   id INT PRIMARY KEY     NOT NULL,
   name           TEXT    NOT NULL,
   age            INT     NOT NULL,
   address        TEXT,
   salary         REAL
);


INSERT INTO company (id, name, age, address, salary)
VALUES 
	(1, 'Paul', 32, 'California', 20000.00 ),
	(2, 'Allen', 25, 'Texas', 15000.00 ),
	(3, 'Teddy', 23, 'Norway', 20000.00 ),
	(4, 'Mark', 25, 'Rich-Mond ', 65000.00 ),
	(5, 'David', 27, 'Texas', 85000.00 ),
	(6, 'Kim', 22, 'South-Hall', 45000.00 ),
	(7, 'James', 24, 'Houston', 10000.00 ),
	(8, 'Paul', 24, 'Houston', 20000.00),
	(9, 'James', 44, 'Norway', 5000.00),
	(10, 'James', 45, 'Texas', 5000.00);
```

此时，COMPANY 表的记录如下：

```postgresql
 id | name  | age |  address   | salary 
----+-------+-----+------------+--------
  1 | Paul  |  32 | California |  20000
  2 | Allen |  25 | Texas      |  15000
  3 | Teddy |  23 | Norway     |  20000
  4 | Mark  |  25 | Rich-Mond  |  65000
  5 | David |  27 | Texas      |  85000
  6 | Kim   |  22 | South-Hall |  45000
  7 | James |  24 | Houston    |  10000
  8 | Paul  |  24 | Houston    |  20000
  9 | James |  44 | Norway     |   5000
 10 | James |  45 | Texas      |   5000
(10 行记录)
```

创建一张 DEPARTMENT 表，添加三个字段：

```postgresql
CREATE TABLE department (
   id INT PRIMARY KEY   NOT NULL,
   dept           TEXT  NOT NULL,
   emp_id         INT   NOT NULL
);
```

向 DEPARTMENT 表插入三条记录：

```postgresql
INSERT INTO department (id, dept, emp_id) 
VALUES 
	(1, 'IT Billing', 1 ),
	(2, 'Engineering', 2 ),
	(3, 'Finance', 7 );
```

## 交叉查询 `CROSS JOIN`

交叉连接返回两个表的笛卡尔积，即左表的每一行与右表的每一行组合。结果行数 = 左表行数 × 右表行数，通常用于生成所有可能的组合

语法格式：

```postgresql
SELECT ... FROM table1 CROSS JOIN table2 ...
```
假设有表 colors（颜色）和 sizes（尺寸）：

```postgresql
CREATE TABLE colors (
	id SERIAL PRIMARY KEY,
  color TEXT NOT NULL
);

CREATE TABLE sizes(
	id SERIAL PRIMARY KEY,
  size TEXT NOT NULL
);

INSERT INTO colors (color)
VALUES
	('红'),
	('蓝');
	
	
INSERT INTO sizes (size)
VALUES
	('S'),
	('M'),
	('L');
```

交叉查询：

```postgresql
SELECT color, size 
FROM colors 
CROSS JOIN sizes 
ORDER BY color;
```

结果示例：

```
 color | size 
-------+------
 红    | S
 红    | M
 红    | L
 蓝    | S
 蓝    | M
 蓝    | L
(6 行记录)
```

## 内连接 `INNER JOIN`

内连接返回两表中匹配的行，非匹配行被过滤掉。是默认且最常用的连接类型

语法格式：

```postgresql
SELECT table1.column1, table2.column2...
FROM table1
[INNER] JOIN table2
ON table1.common_filed = table2.common_field;
```

基于上面的表，我们可以写一个内连接，在 department 表中根据 emp_id 关联查询 company 表中的员工姓名，输出员工姓名及其所属部门， 如下所示：

```postgresql
SELECT c.name, d.dept 
FROM company c
JOIN department d
ON c.id = d.emp_id;
```

结果示例：

```postgresql
 name  |    dept     
-------+-------------
 Paul  | IT Billing
 Allen | Engineering
 James | Finance
(3 行记录)
```

## 左外链接 `LEFT OUTER JOIN`

返回左表所有行 + 右表匹配的行（无匹配则右表字段为 NULL）。常用于“查找左表存在但右表无关联的数据”。

语法格式：

```postgresql
SELECT ... 
FROM table1 
LEFT [OUTER] JOIN table2 
ON conditional_expression ...
```

实例：查询所有员工及其部门（包括无部门的员工）

```postgresql
SELECT e.name, d.dept 
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;
-- 结果示例：
-- name  | dept
-- ------+-----------
-- 张三  | 技术部
-- 王五  | NULL
```

## 右外连接 `RIGHT OUTER JOIN`

语法格式：

```postgresql
SELECT ... 
FROM table1 
RIGHT [OUTER] JOIN table2 
ON conditional_expression ...
```

实例：查询所有部门及其员工（包括无员工的部门）

```postgresql
SELECT e.name, d.dept 
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.id;
-- 结果示例：
-- name  | dept
-- ------+-----------
-- 张三  | 技术部
-- NULL  | 财务部
```

## 全外链接 `FULL OUTER JOIN`

语法格式：

```postgresql
SELECT ... 
FROM table1 
FULL [OUTER] JOIN table2 
ON conditional_expression ...
```

实例：合并员工和部门信息（保留所有记录）

```postgresql
SELECT e.name, d.dept 
FROM employees e
FULL JOIN departments d ON e.dept_id = d.id;
-- 结果示例：
-- name  | dept
-- ------+-----------
-- 张三  | 技术部
-- 王五  | NULL
-- NULL  | 财务部
```

# 操作符 `UNION`

## 概述

@See https://www.runoob.com/postgresql/postgresql-union.html

`UNION` 用于合并两个或多个 `SELECT` 语句的结果集，**自动去除重复行**。若需保留重复行，需使用 `UNION ALL`。

**核心规则**：

- 所有 `SELECT` 语句的列数必须相同。
- 对应列的数据类型必须兼容（如 `INT` 和 `BIGINT` 兼容，但 `TEXT` 和 `INT` 不兼容）

**基础语法**：

```postgresql
SELECT column1 [, column2 ]
FROM table1 [, table2 ]
[WHERE condition]

UNION

SELECT column1 [, column2 ]
FROM table1 [, table2 ]
[WHERE condition]
```

## 创建测试表与数据

**表1：`employees`（正式员工）**

```postgresql
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    department VARCHAR(20)
);

INSERT INTO employees (id, name, department) 
VALUES 
    (1, '张三', '技术部'),
    (2, '李四', '人事部'),
    (3, '王五', '财务部');
```

**表2：`contractors`（外包员工）**

```postgresql
CREATE TABLE contractors (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    department VARCHAR(20)
);

INSERT INTO contractors (id, name, department) 
VALUES 
    (4, '赵六', '技术部'),
    (5, '李四', '人事部'),  -- 重复姓名
    (6, '钱七', '市场部');
```

## 实例演示

#### 基本 UNION（去重）

合并所有员工姓名和部门，自动去除重复行（如“李四”仅出现一次）：

```postgresql
SELECT name, department FROM employees
UNION
SELECT name, department FROM contractors;
```

结果示例：

```postgresql
 name | department 
------+------------
 赵六 | 技术部
 钱七 | 市场部
 张三 | 技术部
 李四 | 人事部
 王五 | 财务部
(5 行记录)
```

#### UNION ALL（保留重复）

保留所有记录，包括重复的“李四”：

```postgresql
SELECT name, department FROM employees
UNION ALL
SELECT name, department FROM contractors;
```

结果示例：

```postgresql
 name | department 
------+------------
 张三 | 技术部
 李四 | 人事部
 王五 | 财务部
 赵六 | 技术部
 李四 | 人事部
 钱七 | 市场部
(6 行记录)
```

#### 多表合并与排序

合并三张表（假设新增 `interns` 表）并按姓名排序：

```postgresql
SELECT name, department FROM employees
UNION ALL
SELECT name, department FROM contractors
UNION ALL
SELECT name, department FROM interns
ORDER BY name;  -- 排序必须放在最后
```

## 常见应用场景

1. **数据整合**：合并结构相同的表（如分表存储的历史数据）
2. **汇总统计**：计算多表的总和或平均值（如各区域销售总额）
3. **数据比对**：查找不同表中的差异记录（结合 `EXCEPT` 或 `INTERSECT`）

# NULL

@See https://www.runoob.com/postgresql/postgresql-null.html

NULL 值代表遗漏的未知数据。默认地，表的列可以存放 NULL 值。

**语法格式**：

当创建表时，NULL 的基本语法如下：

```postgresql
CREATE TABLE company(
   id INT PRIMARY KEY     NOT NULL,
   name           TEXT    NOT NULL,
   age            INT     NOT NULL,
   address        TEXT,
   salary         REAL
);
```

这里，NOT NULL 表示强制字段始终包含值。这意味着，如果不向字段添加值，就无法插入新记录或者更新记录。

具有 NULL 值的字段表示在创建记录时可以留空。

在查询数据时，NULL 值可能会导致一些问题，因为一个未知的值去与其他任何值比较，结果永远是未知的。

另外无法比较 NULL 和 0，因为它们是不等价的。

**实例**

创建 COMPANY 表，数据内容如下：

```postgresql
INSERT INTO company (id, name, age, address, salary)
VALUES
	(1, 'Paul', 32, 'California', 20000.00 ),
	(2, 'Allen', 25, 'Texas', 15000.00 ),
	(3, 'Teddy', 23, 'Norway', 20000.00 ),
	(4, 'Mark', 25, 'Rich-Mond ', 65000.00 ),
	(5, 'David', 27, 'Texas', 85000.00 ),
	(6, 'Kim', 22, 'South-Hall', 45000.00 ),
	(7, 'James', 24, 'Houston', 10000.00 );
	
SELECT * FROM company;
 id | name  | age |  address   | salary 
----+-------+-----+------------+--------
  1 | Paul  |  32 | California |  20000
  2 | Allen |  25 | Texas      |  15000
  3 | Teddy |  23 | Norway     |  20000
  4 | Mark  |  25 | Rich-Mond  |  65000
  5 | David |  27 | Texas      |  85000
  6 | Kim   |  22 | South-Hall |  45000
  7 | James |  24 | Houston    |  10000
(7 行记录)
```

# 索引

@See https://www.runoob.com/postgresql/postgresql-index.html

索引是加速搜索引擎检索数据的一种特殊表查询。简单地说，索引是一个指向表中数据的指针。一个数据库中的索引与一本书的索引目录是非常相似的。

拿汉语字典的目录页（索引）打比方，我们可以按拼音、笔画、偏旁部首等排序的目录（索引）快速查找到需要的字。

索引有助于加快 SELECT 查询和 WHERE 子句，但它会减慢使用 UPDATE 和 INSERT 语句时的数据输入。索引可以创建或删除，但不会影响数据。

使用 CREATE INDEX 语句创建索引，它允许命名索引，指定表及要索引的一列或多列，并指示索引是升序排列还是降序排列。

索引也可以是唯一的，与 UNIQUE 约束类似，在列上或列组合上防止重复条目。

## CREATE INDEX

CREATE INDEX （创建索引）的语法如下：

```postgresql
CREATE INDEX index_name ON table_name;
```

## 索引类型

### 单列索引

单列索引是一个只基于表的一个列上创建的索引，基本语法如下：

```postgresql
CREATE INDEX index_name
ON table_name (column_name);
```

### 组合索引

组合索引是基于表的多列上创建的索引，基本语法如下：

```postgresql
CREATE INDEX index_name
ON table_name (column1_name, column2_name);
```

不管是单列索引还是组合索引，该索引必须是在 WHERE 子句的过滤条件中使用非常频繁的列。

如果只有一列被使用到，就选择单列索引，如果有多列就使用组合索引。

### 唯一索引

使用唯一索引不仅是为了性能，同时也为了数据的完整性。唯一索引不允许任何重复的值插入到表中。基本语法如下：

```postgresql
CREATE UNIQUE INDEX index_name
on table_name (column_name);
```

### 局部索引

局部索引 是在表的子集上构建的索引；子集由一个条件表达式上定义。索引只包含满足条件的行。基础语法如下：

```postgresql
CREATE INDEX index_name
ON table_name(column_list)
WHERE condition;
```

在这里，index_name 是你想要创建的索引的名称，table_name 是包含你想要索引的列的表的名称，column_list 是你想要索引的列的列表，而 condition 是一个布尔表达式，用于定义哪些行将被包含在索引中。

### 隐式索引

在 PostgreSQL 中，隐式索引是在创建对象时，由数据库服务器自动创建的索引。这类索引通常为主键约束和唯一约束自动创建。当在创建表时声明一个列为主键、唯一约束或外键时，PostgreSQL 会自动为该列创建一个隐式索引。这样做的好处是简化了索引管理，并且提高了数据库的性能。

例如，如果在创建一个名为 "users" 的表时，声明了一个名为 "userid" 的列为主键，PostgreSQL会自动为 "userid" 列创建一个隐式索引，这意味着在插入新记录时，数据库会自动为 "userid" 列生成一个唯一的索引值。

隐式索引的创建和管理是由 PostgreSQL 自动完成的，用户不需要手动干预，这使得数据库管理变得更加简单和高效。

## 实例

下面实例将在 COMPANY 表的 SALARY 列上创建索引：

```postgresql
CREATE INDEX salary_index ON company(salary);
```

现在，用 **\d company** 命令列出 COMPANY 表的所有索引：

```postgresql

    "company_pkey" PRIMARY KEY, btree (id)

```

```postgresql

```


## 什么情况下要避免使用索引？

虽然索引的目的在于提高数据库的性能，但这里有几个情况需要避免使用索引。

使用索引时，需要考虑下列准则：

- 索引不应该使用在较小的表上。
- 索引不应该使用在有频繁的大批量的更新或插入操作的表上。
- 索引不应该使用在含有大量的 NULL 值的列上。
- 索引不应该使用在频繁操作的列上。

# TRANSACTION

@See https://www.runoob.com/postgresql/postgresql-transaction.html

事务是 PostgreSQL 中逻辑操作的最小单元，由一组原子性的数据库操作（如 INSERT/UPDATE/DELETE）组成，核心目标是确保数据在并发和异常场景下的**一致性**与**可靠性**。其设计解决了两个关键问题：

1. **故障恢复**：事务失败时自动回滚到操作前状态，避免数据部分更新
2. **并发隔离**：通过多版本并发控制（MVCC）机制，使并发事务相互隔离，避免读写冲突。

## 事务的 ACID 属性

| 属性                     | PostgreSQL 实现机制                                          | 实际场景示例                                                 |
| :----------------------- | :----------------------------------------------------------- | ------------------------------------------------------------ |
| **原子性** (Atomicity)   | 事务内操作要么全部提交（通过 WAL 日志持久化），要么全部回滚（撤销所有未提交的变更）。 | 转账事务：扣款和到账操作必须同时成功，否则回滚至初始状态。   |
| **一致性** (Consistency) | 通过约束（如 CHECK、外键）和触发器保证数据修改符合业务规则，即使事务中断也保持约束成立。 | 账户余额不允许为负值，事务若导致负余额会被强制回滚。         |
| **隔离性** (Isolation)   | 默认使用 **Read Committed** 隔离级别，基于 MVCC 实现读写不阻塞，通过快照避免脏读。 | 事务A读取数据时，事务B可同时修改同一数据，A看到的是修改前的快照版本。 |
| **持久性** (Durability)  | 依赖预写式日志（WAL）：事务提交前先将变更写入磁盘日志，崩溃后可通过日志恢复。 | 即使数据库崩溃，已提交的转账记录不会丢失。                   |

## 事务控制命令

### 1. 开始事务

```postgresql
BEGIN;  -- 或 BEGIN TRANSACTION;
```

- **作用**：显式开启事务，后续操作暂不持久化，直到提交或回滚
- **注意**：DDL（如 CREATE TABLE）会自动提交，无法包含在事务中

### 2. 提交事务

```postgresql
COMMIT;  -- 或 COMMIT AND CHAIN（提交后立即启动新事务，减少网络往返）
```

- **效果**：确认所有变更，写入磁盘并释放锁

### 3. 回滚事务

```postgresql
ROLLBACK;  -- 或 ROLLBACK TO SAVEPOINT（部分回滚到保存点）
```

- **场景**：事务执行出错时撤销全部操作，或通过保存点（SAVEPOINT）实现局部回滚

## 实战示例与优化技巧

1. **基础事务流程**

   ```postgresql
   BEGIN;
   UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;  -- 扣款  
   UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;  -- 到账  
   COMMIT;  -- 若任一UPDATE失败，自动回滚
   ```

2. **高性能事务设计**

   - **减小事务粒度**：拆分大事务为多个小事务，减少锁竞争和冲突（如订单处理拆分为插入订单、更新库存等独立事务）。
   - **链式事务**：使用 `COMMIT AND CHAIN` 减少客户端与服务器的通信开销，适合高频短事务。
   - **WAL 优化**：调整 `commit_delay` 和 `commit_siblings` 参数，合并 WAL 刷写以提升吞吐量（需权衡延迟）

3. **并发控制**

   - **隔离级别选择**：

     - **Read Committed**（默认）：平衡性能与一致性，允许不可重复读和幻读。
     - **Repeatable Read**：通过快照避免不可重复读，适合需数据稳定性的场景（如报表生成）

   - **锁机制**：

     ```postgresql
     SELECT * FROM orders WHERE id = 100 FOR UPDATE;  -- 行锁防止其他事务修改
     ```

## 常见问题与解决方案

- **死锁**：PostgreSQL 自动检测并回滚冲突事务，建议按固定顺序访问资源（如先锁表A再锁表B）。
- **长事务阻塞**：监控 `pg_stat_activity`，终止长时间运行的事务。

# LOCK

@See https://www.runoob.com/postgresql/postgresql-lock.html

锁是 PostgreSQL 实现并发控制的关键组件，通过多粒度锁（表级、行级、页级）和冲突检测机制，在 MVCC 基础上保障数据一致性。其核心价值体现在：

1. **并发协调**：允许读写操作在隔离级别约束下并行执行（如 `READ COMMITTED` 模式下读不阻塞写）。
2. **数据完整性**：防止并发事务导致的数据覆盖或逻辑冲突（如丢失更新、脏读）。
